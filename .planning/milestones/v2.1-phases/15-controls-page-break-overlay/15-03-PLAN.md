**PDF export path:**
The PDF export creates a hidden BrowserWindow that loads print.html. The template handles its own padding (margins are inline styles on the template wrapper). The `printToPDF` call already uses zero margins (`{ top: 0, bottom: 0, left: 0, right: 0 }`) because the template controls all spacing. This remains correct — no change to the printToPDF margins parameter.

However, the PDF export BrowserWindow's PrintApp needs to know the templateOptions so it can pass margin/accent/skills values to the template. In Plan 02 Task 2, PrintApp's BrowserWindow path already calls `window.api.templates.getOptions(variantId)`. So the PDF path is already handled by Plan 02. **Verify this is the case** — if PrintApp's BrowserWindow code path does not yet call getOptions, add it here.

**DOCX export path:**
1. Read `templateOptions` from the `templateVariants` row. The variant row is already queried in the DOCX handler (to get `layoutTemplate` for font selection). Add reading `templateOptions` from the same row.
2. Parse `templateOptions` as JSON (with try/catch, default to `{}`).
3. Import `TEMPLATE_DEFAULTS` — since this is main process code and TEMPLATE_DEFAULTS is in a renderer file, define the same defaults inline in export.ts (or extract to a shared constants file). Simplest: define `DOCX_MARGIN_DEFAULTS` inline in export.ts matching the TEMPLATE_DEFAULTS structure.
4. Compute margin values in twips (1 inch = 1440 twips):
   ```typescript
   const defaults = DOCX_MARGIN_DEFAULTS[layoutTemplate] ?? { top: 1.0, bottom: 1.0, sides: 1.0 }
   const mt = Math.round((templateOptions.marginTop ?? defaults.top) * 1440)
   const mb = Math.round((templateOptions.marginBottom ?? defaults.bottom) * 1440)
   const ms = Math.round((templateOptions.marginSides ?? defaults.sides) * 1440)
   ```
5. Replace the hardcoded `page: { margin: { top: 720, bottom: 720, left: 720, right: 720 } }` with: `page: { margin: { top: mt, bottom: mb, left: ms, right: ms } }`.
6. Also read `skillsDisplay` from templateOptions and pass it to the DOCX generation logic if it affects section rendering (currently DOCX renders skills grouped by default — if `skillsDisplay === 'inline'`, render as comma-separated list instead of category groups). Check current DOCX skills rendering and adapt if needed.
7. Read `accentColor` from templateOptions — currently DOCX does not use accent color (Word docs are black/white), so no change needed for accent. Just note this in comments.

**DOCX_MARGIN_DEFAULTS constant:**
