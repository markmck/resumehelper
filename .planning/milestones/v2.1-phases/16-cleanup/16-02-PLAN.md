**PrintApp.tsx — Add snapshot mode:**

In the main useEffect that fetches data, add a snapshot mode check. The existing code bifurcates on `window.api` availability (BrowserWindow vs iframe). Add a third path for snapshot mode:

1. Read `variantId` from URL search params (already done in existing code)
2. Add: `const isSnapshotMode = variantId === 0` — this is the sentinel value indicating snapshot data will arrive via postMessage
3. Modify the conditional logic:
   - If `!isSnapshotMode && window.api is available` → existing BrowserWindow fetch-by-variantId path (unchanged)
   - If `isSnapshotMode || window.api is not available` → postMessage listener path (this combines iframe mode and snapshot BrowserWindow mode)
4. In the postMessage path, when `isSnapshotMode` AND `window.electron` is available, send `print:ready` via `window.electron.ipcRenderer.send('print:ready')` so the main process knows to push snapshot data. This must fire BEFORE data arrives (to trigger the executeJavaScript in the handler).
5. After receiving `print-data` postMessage and setting state, the existing `print:ready` IPC send (around line 221) will fire again — this is the signal that rendering is complete. The snapshotPdf handler uses a settle delay after executeJavaScript rather than waiting for this second signal, so no change needed there.

Key details:
- The `template` query param from the URL should be read and used as the template override if present (for snapshot mode, the handler sets `template=resolvedTemplate` in the URL)
- The `print-data` postMessage payload includes `template`, `showSummary`, and `payload` (the resume data) — this is the same structure already used by VariantPreview iframe mode

**export.ts — Rewrite snapshotPdf handler:**

Replace the existing `export:snapshotPdf` handler (which uses the now-deleted renderThemeHtml) with a print.html BrowserWindow pipeline. Follow the pattern from the research document's code example:

4. Create BrowserWindow with `show: false`, `width: 816`, `height: 1056`, webPreferences with preload path and `sandbox: false` (preload needed for print:ready IPC signal).

5. Load print.html URL with query params `variantId=0&template={resolvedTemplate}` — use the same dev/prod URL construction as the existing export:pdf handler.

6. Wait for `ipcMain.once('print:ready', ...)` — then push data via `win.webContents.executeJavaScript(...)` using `window.postMessage({ type: 'print-data', template, showSummary: true, payload }, '*')`.
