<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| renderer → main (IPC `analysisLayout:*`) | Untrusted-shaped args (analysisId, margin triple) cross from renderer into main-process DB writes. Local desktop app; values originate from trusted local UI in later phases, but the handler must not assume well-formed input. |
| main → SQLite | DB write of analysis-scoped margin rows. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-39-01 | Tampering | `setAnalysisMargins` margin args | mitigate | Reject non-finite numbers (NaN/Infinity) via `Number.isFinite` guard before any write (D-09). No bounds/floor (owned by Phase 40/41 UI). |
| T-39-02 | Tampering | SQL for the new table | mitigate | All reads/writes use Drizzle bound parameters (`eq`, `.values`, `.onConflictDoUpdate`) — no string interpolation of IDs or margins. |
| T-39-03 | Information Disclosure / Integrity | FK `analysis_id` → `analysis_results` ON DELETE CASCADE | mitigate | UNIQUE + CASCADE FK so an override is scoped to exactly one live analysis and is auto-removed on analysis delete (no orphan rows). Cascade asserted in unit test. |
| T-39-04 | Denial of Service | repeated set on same analysisId | accept | `onConflictDoUpdate` keeps it to one row per analysis; unbounded distinct analyses is inherent to the app and low-value locally. |
</threat_model>
