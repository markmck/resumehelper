          )

          return { canceled: true, error: 'validation' as const }
        }
        throw err

      const { canceled, filePath } = await dialog.showSaveDialog({

    })

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| renderer → preload → main IPC | Renderer calls `window.api.exportFile.json(defaultFilename)`. `defaultFilename` is constructed in renderer from `sanitizeFilename(profileData.name || 'Resume')` — already sanitized to `[a-zA-Z0-9_-]` charset. |
| main → OS save dialog → user → filesystem | User chooses save location. Electron's `showSaveDialog` returns an OS-validated absolute path. Main then `fs.writeFile`s to that path. |
| main → SQLite | `setSetting(db, 'lastExportDir', dirname(filePath))` writes the user-chosen path back to the DB via Drizzle parameterization. |
| main → native error dialog | `dialog.showErrorBox(title, body)` — first use site in codebase per RESEARCH.md Pitfall 1. Title and body are constant strings + interpolated Zod issue paths (already string-typed). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-31-11 | Tampering — path traversal via save dialog | `fs.writeFile(filePath, ...)` where `filePath` came from `dialog.showSaveDialog` | mitigate | Electron's `showSaveDialog` returns an OS-validated absolute path the user explicitly chose. The renderer cannot influence the returned `filePath` (it's main-process trusted output). Path-traversal sequences in `defaultFilename` would be neutralized by the user re-selecting in the dialog anyway, but additionally `defaultFilename` is constructed via `sanitizeFilename` which strips `/`, `\`, `.`, `:` — so no traversal payload can reach the dialog as a default. Verified by reading sanitizeFilename regex: `[^a-zA-Z0-9_-]` removed → dots and slashes stripped. |
| T-31-12 | Tampering — arbitrary file overwrite | save dialog allowing user to overwrite an existing file | accept | OS save dialog confirms overwrite natively. User-initiated, single-user app. No mitigation needed beyond OS-native behavior. |
| T-31-13 | Injection — Zod issue paths in error dialog body | `dialog.showErrorBox(title, body)` with interpolated issue paths | mitigate | `dialog.showErrorBox` renders body as plain text (not HTML) per Electron docs. No XSS / injection surface. Worst case: a malicious DB value produces a long ugly path string — annoying, not harmful. Truncated to 5 issues to bound size. |
| T-31-14 | Information Disclosure — lastExportDir leak | `lastExportDir` value stored in DB | accept | The path is user-chosen; storing it back is no different from any "recent files" UX. Local app, local DB. |
| T-31-15 | Information Disclosure — sensitive data in exported JSON | `basics.email`, `basics.phone`, `linkedin`, work history | accept | User initiated the export. The exported JSON contains only what the user already entered. JSDoc (Plan 02 D-16) documents which fields are exported so the user can audit. No automatic export, no telemetry, no cloud upload. |
| T-31-16 | Repudiation — silent write of invalid data | Plan 02's ExportValidationError gate | mitigate | D-13 + D-15: handler validates BEFORE save dialog. No save dialog appears on validation failure. No file is ever written for invalid data. User sees explicit error dialog naming the issue paths. Verified by acceptance criteria on Task 1 (validation-first ordering). |
| T-31-17 | Denial of Service — runaway Zod issues list | issues.slice(0, 5) | mitigate | Slice caps the preview at 5 even if Zod reports thousands of issues. Full list written to console.error (not user-blocking). |
| T-31-18 | Schema injection via app_settings k/v | Plan 03 writes `dirname(filePath)` to settings | mitigate | Drizzle parameterizes the UPSERT (verified in Plan 01 T-31-03). `dirname()` returns a plain string; it cannot contain SQL metacharacters that would escape Drizzle's prepared statement. |

No `high` severity threats. The primary risk surfaces (path traversal, silent invalid writes, injection) are all neutralized by validation-first ordering + sanitizeFilename + Drizzle parameterization + Electron's native dialog contract.
