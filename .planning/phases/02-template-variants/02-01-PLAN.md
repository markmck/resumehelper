1. **Modify `src/main/db/schema.ts`** — update the two template tables:

   `templateVariants`: Add `layoutTemplate: text('layout_template').notNull().default('traditional')` column after `name`.

   `templateVariantItems`: Replace entire table definition with the discriminator model:
   - Keep `id` (autoIncrement PK) and `variantId` (FK to templateVariants with cascade delete)
   - Add `itemType: text('item_type').notNull()` — values: 'bullet' | 'skill' | 'job'
   - Change `bulletId` to nullable: `integer('bullet_id').references(() => jobBullets.id, { onDelete: 'cascade' })`
   - Add `skillId: integer('skill_id').references(() => skills.id, { onDelete: 'cascade' })`
   - Add `jobId: integer('job_id').references(() => jobs.id, { onDelete: 'cascade' })`
   - Replace `included` with `excluded: integer('excluded', { mode: 'boolean' }).notNull().default(false)`
   - The explicit-exclusion model means: no row = item is included; row with excluded=true = item is excluded. This avoids backfill when new experience items are added.

2. **Generate migration** — run `npx drizzle-kit generate` from project root. This produces a new SQL file in `drizzle/`. Verify the migration file exists and contains ALTER TABLE or CREATE TABLE statements for the schema changes.

<task type="auto">
  <name>Task 2: Template IPC handlers</name>
  <files>src/main/handlers/templates.ts, src/main/handlers/index.ts</files>
  <action>
1. **Create `src/main/handlers/templates.ts`** with a `registerTemplateHandlers()` export. Follow the exact pattern from `src/main/handlers/jobs.ts` — import `ipcMain`, `db`, schema tables, and drizzle operators.

   Implement these handlers:

   **`templates:list`** — `db.select().from(templateVariants).orderBy(desc(templateVariants.createdAt))`. Return all variants.

   **`templates:create`** — Insert with `{ name: data.name }` (layoutTemplate defaults to 'traditional'). Return the new row via `.returning()`.

   **`templates:rename`** — Update `name` where `id = variantId`. Return updated row.

   **`templates:delete`** — Delete where `id = variantId`. CASCADE handles templateVariantItems cleanup.

   **`templates:duplicate`** — Wrap in a better-sqlite3 transaction (access raw sqlite via `db.$client`):
   - Select source variant by id
   - Insert new variant with name `"${source.name} (Copy)"` and same `layoutTemplate`
   - Select all templateVariantItems where variantId = sourceId
   - Insert copies with the new variantId
   - Return the new variant
   - Use `db.transaction(async (tx) => { ... })` for the atomic operation

   **`templates:setLayoutTemplate`** — Update `layoutTemplate` where `id = variantId`. Return updated row.

   **`templates:getBuilderData`** — This is the key query:
   - Fetch all jobs ordered by createdAt desc
   - For each job, fetch its bullets ordered by sortOrder asc
   - Fetch all skills
   - Fetch all templateVariantItems where variantId = given id
   - Build exclusion lookup Sets: excludedJobIds, excludedBulletIds, excludedSkillIds
   - Return `{ jobs: [...with excluded flag and bullets with excluded flag], skills: [...with excluded flag] }`
   - Parse skills.tags from JSON string to string[] (same pattern as skills handler)

   **`templates:setItemExcluded`** — Toggle exclusion for a single item:
   - If `excluded` is true: insert a row with `{ variantId, itemType, bulletId/skillId/jobId (based on itemType), excluded: true }`. Use delete-then-insert to avoid duplicate row issues (no unique constraint needed).
   - If `excluded` is false: delete the row matching `variantId + itemType + corresponding id column`.
   - When itemType is 'job' and excluded is true: also insert exclusion rows for all of that job's bullets (fetch bullet ids first). When itemType is 'job' and excluded is false: also delete exclusion rows for all of that job's bullets. This implements the "checking/unchecking a job toggles all its bullets" requirement.
