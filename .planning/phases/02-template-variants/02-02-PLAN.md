2. **Create `src/renderer/src/components/TemplatesTab.tsx`:**
   - Two-column layout: fixed-width sidebar (w-64) + flex-1 editor area
   - State: `variants: TemplateVariant[]`, `selectedId: number | null`
   - On mount: fetch variants via `window.api.templates.list()`, auto-select first if any
   - Handlers:
     - `handleCreate`: calls `window.api.templates.create({ name: 'Untitled Variant' })`, adds to list, selects the new variant
     - `handleDelete`: calls `window.api.templates.delete(id)`, removes from list, selects next available or null
     - `handleDuplicate`: calls `window.api.templates.duplicate(id)`, adds copy to list, selects the copy (per user decision: "duplicating opens the copy immediately")
     - `handleRename`: calls `window.api.templates.rename(id, newName)`, updates in list
   - Render: `<VariantList>` in sidebar, `<VariantEditor>` or empty state in editor area
   - Empty state when no variants: centered message with "Create your first variant" button
   - Match dark/minimal zinc aesthetic from Phase 1

3. **Create `src/renderer/src/components/VariantList.tsx`:**
   - Props: `variants`, `selectedId`, `onSelect`, `onCreate`, `onDelete`, `onDuplicate`
   - Renders a scrollable list of variant names in the sidebar
   - Each variant row: click to select (highlighted with bg-zinc-800 or similar)
   - Group/group-hover pattern for contextual controls: duplicate and delete buttons appear on row hover (same pattern as Phase 1 delete buttons)
   - "New Variant" button at the top of the sidebar (+ icon)
   - Immediate delete, no confirmation dialog (per user decision)

4. **Create `src/renderer/src/components/VariantEditor.tsx`:**
   - Props: `variant: TemplateVariant`, `onRename: (id, name) => void`
   - State: `activeSubTab: 'builder' | 'preview'` (default 'builder')
   - Top area: variant name rendered with `<InlineEdit>` for click-to-rename (reuse from Phase 1)
   - Layout template selector: a simple dropdown or button group showing available templates ('traditional', 'modern', 'compact'). When changed, call `window.api.templates.setLayoutTemplate(variant.id, newTemplate)` and update parent state.
   - Sub-tab bar below name: "Builder" and "Preview" buttons
   - Content area: conditionally render `<VariantBuilder>` or `<VariantPreview>` based on activeSubTab

5. **Create `src/renderer/src/components/VariantBuilder.tsx`:**
   - Props: `variantId: number`
   - State: `builderData: BuilderData | null`
   - On mount (and when variantId changes): fetch via `window.api.templates.getBuilderData(variantId)`
   - Render two sections:

   **Work History section:**
   - Each job rendered as a card/group with a checkbox next to `company — role`
   - Below each job: indented list of bullets, each with its own checkbox
   - Job checkbox: when toggled, calls `window.api.templates.setItemExcluded(variantId, 'job', jobId, !currentExcluded)`. Optimistic update: set job.excluded locally. When a job is excluded, all its bullets render as disabled/grayed out (visually subordinate to the parent toggle).
   - Bullet checkbox: only interactive when parent job is NOT excluded. When toggled, calls `window.api.templates.setItemExcluded(variantId, 'bullet', bulletId, !currentExcluded)`. Optimistic update in local state.
   - Individual bullets can be toggled independently within an included job (per user decision).

   **Skills section:**
   - Group skills by their tags (same grouping pattern as ExperienceTab's SkillList)
   - Each skill has a checkbox. When toggled, calls `window.api.templates.setItemExcluded(variantId, 'skill', skillId, !currentExcluded)`. Optimistic update.

   - All changes persist immediately to database (no save button) per user decision.
   - Use standard HTML `<input type="checkbox">` — no library needed.

6. **Create `src/renderer/src/components/VariantPreview.tsx`:**
   - Props: `variantId: number`, `layoutTemplate: string`
   - State: `builderData: BuilderData | null`
   - On mount: fetch via `window.api.templates.getBuilderData(variantId)` (same data as builder, filtered to included items only)
   - Render a resume-like read-only view showing only the INCLUDED items (where excluded !== true):
     - For 'traditional' layout: classic resume format — name/header area (placeholder), Work Experience section with job titles and bullet points, Skills section grouped by tags
     - For 'modern' layout: two-column layout — skills sidebar on left, work experience on right
     - For 'compact' layout: condensed single-column, smaller text, tighter spacing
   - Use Tailwind classes for styling. This is a preview within the app window, not a separate render path.
   - Section order is driven by the layout template selection, not manual reordering (per user decision).
   - No editing in preview — purely read-only display.
