## Spacing Scale

Declared values — all from `--space-*` tokens in `tokens.css`:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px (`--space-1`) | Icon gaps, badge padding |
| sm | 8px (`--space-2`) | Compact element spacing, button icon gaps |
| md | 16px (`--space-4`) | Default element spacing, card inner padding |
| lg | 24px (`--space-6`) | Section padding, group spacing |
| xl | 32px (`--space-8`) | Layout gaps between major sections |
| 2xl | 48px (`--space-12`) | Major section breaks |
| 3xl | 64px (`--space-16`) | Page-level spacing |

Exceptions: `--space-3` (12px) and `--space-5` (20px) exist in the token scale and are used for intermediate spacing (e.g. toast padding) — follow the same pattern where tight fit requires it.

Touch targets: per-bullet action buttons (Accept, Revert) use `padding: 4px 12px` — this is a desktop-only Electron app with no mobile touch constraints.

## Color

All values from `tokens.css`. Phase 18 uses the existing dark-surface palette.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#0a0a0b` (`--color-bg-base`) | Page background |
| Secondary (30%) | `#141416` / `#1c1c1f` (`--color-bg-surface` / `--color-bg-raised`) | Card backgrounds, suggestion rows, section containers |

|-------|-------|--------------|
| `--color-success` | `#22c55e` | Accepted-state inline label ("Accepted") |
| `--color-warning` | `#f59e0b` | Point-impact badge on moderate suggestions |

## Interaction Contract

### Accept Flow (per-bullet)

1. User clicks "Accept" button on a pending suggestion.
2. Button enters a loading/disabled state (disable the button immediately to prevent double-click).
3. `await window.api.ai.acceptSuggestion(analysisId, bulletId, finalText)` is called.
4. On success: suggestion state transitions to `'accepted'`. Revert button appears. Toast fires "Suggestion accepted".
5. `previewRefreshKey` increments — VariantPreview re-fetches with analysisId, preview updates.
6. On error: console.error, button re-enables, no state change, no toast.

### Revert Flow (per-bullet)

1. User clicks "Revert" on an accepted suggestion.
2. `await window.api.ai.dismissSuggestion(analysisId, bulletId)` is called.
3. On success: suggestion state resets to `'pending'` with original `suggested_text` as `finalText`. Revert button hides. Accept button reappears. Toast fires "Suggestion reverted".
4. `previewRefreshKey` increments.

### Accept-All Flow

1. User clicks "Accept all" (if button exists in current UI).
2. All pending suggestions fire `ai:acceptSuggestion` in parallel via `Promise.all`.
3. After all resolve: all accepted states update in a single `setSuggStates` call. One `previewRefreshKey` increment. Toast fires "All suggestions accepted".
4. On partial failure: log errors for failed items, update only the succeeded items in state.

### Preview Refresh

- `VariantPreview` receives `analysisId` prop when rendered inside `OptimizeVariant`.
- `getBuilderData(variantId, analysisId)` is called on each `refreshKey` change.
- No optimistic local text update — re-fetch drives truth (CONTEXT.md D-06).
- Preview iframe performs a full reload on each refresh (~200ms) — this is acceptable per D-06.

## States to Render

Each suggestion row must handle all four states:

| State | Visible Elements |
|-------|-----------------|
| `pending` | Original text (muted), suggested text (primary), Accept button, Skip/Edit controls |
| `accepted` | Suggested text (primary), "Accepted" badge (success color), Revert button |
| `dismissed` | Original text (muted), "Skipped" label (muted) — existing pattern, no change |
| `loading` | Accept/Revert button disabled, no spinner needed (IPC is fast SQLite write) |

Each skill row must handle three states:

| State | Visible Elements |
|-------|-----------------|
| `pending` | Skill name, reason/severity chip, Add button, Skip button |
| `accepted` | Skill name, "Added to analysis" label (success color) |
| `dismissed` | Skill name, "Skipped" label (muted color) |
